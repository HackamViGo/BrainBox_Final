# Core Rules: Documentation

## 1. Архитектурни решения (ADR)

Файл: `docs/DECISIONS.md`.

**Пиши в него при ВСЯКО структурно решение.**

Формат: `## [ADR-NNN] Заглавие` + Контекст, Решение, Причини, Последствия.

## 2. README.md

Всеки package и app трябва да има `README.md` с:

- Setup инструкции.
- Как се стартира локално.
- Описание на публичните API-та.

## 3. Inline документация

- **JSDoc**: Задължителен за всички публични функции, Server Actions и Zustand stores.
- **`@throws`**: Маркирай възможните AppErrors.
- **Animation Components**: Описвай всеки `useEffect` dependency.
- **Сложна логика**: Обяснявай **ЗАЩО** (бизнес контекст), а не КАКВО (кодът го показва).

## 4. Графи на зависимости (AGENTS_GRAPH.json)

- Обновявай `docs/AGENTS_GRAPH.json` при добавяне на нови файлове в `.agents/`.
- Поддържай `status: "active"` или `"deprecated"`.
- `responsibility` трябва да е точно едно изречение.

## 5. Master Prompt

- `docs/BRAINBOX_V2_MASTER_PROMPT_FINAL.md` е живата спецификация на проекта.
- Обновявай фазите на развитие при завършване на голям етап.
- Ако Липсва информация или има грешно записана такава. Спри и веднага съобщи на потребителя. Опиши проблема/пропуска, дай предложение за решеването му, след като си проверил в context7 (MCP, което ползваш за актуална информация, ако е изкл. Задължително вкл)
